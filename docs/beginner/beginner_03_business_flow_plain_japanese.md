# 初心者向け：今回の業務フローを人間の言葉で理解する

## 業務フローとは何か

業務フローとは、業務の流れを図にしたものです。

標準仕様書の別紙1には、BPMNという形式で業務フローが書かれています。BPMNは、業務の開始、作業、判断、システム処理、帳票、終了などを図で表す方法です。

ただし、初心者にはかなり読みにくいです。

ここでは、今回のデモに必要な業務フローだけを、普通の文章で説明します。

---

# 02-01 住民情報異動等に伴う資格異動

## 何の業務か

これは、

> 住民の情報が変わったときに、介護保険の資格情報も更新する業務

です。

住民情報の変化には、以下があります。

| 住民側の変化 | 介護保険側の処理 |
|---|---|
| 65歳になった | 資格取得 |
| 転入した | 資格取得 |
| 転出した | 資格喪失 |
| 死亡した | 資格喪失 |
| 住所が変わった | 資格情報変更 |
| 氏名が変わった | 資格情報変更 |

---

## 誰が関係するか

標準の業務フローでは、主に次の人・部署が出てきます。

| 登場人物・部署 | 初心者向け説明 |
|---|---|
| 住民 | 市民本人 |
| 住民記録担当課 | 住民票や住所異動を管理する部署 |
| 介護保険担当課 | 介護保険業務を担当する部署 |
| 介護保険システム | 今回作る対象のシステム |

今回のPoCでは、住民記録担当課や住民記録システムは本物としては登場しません。CSVや手入力で代替します。

---

## 本来の流れ

標準仕様のイメージを文章にすると、以下です。

```text
1. 住民記録側で住民情報に変化が起きる
2. 介護保険担当課が住民異動情報を受け取る
3. 介護保険システムで対象者を確認する
4. 必要に応じて資格情報を登録・変更・喪失する
5. 必要に応じて被保険者証や受給資格証明書を作成する
6. 証を住民へ送付または手渡しする
```

---

## 今回のデモでの流れ

今回のデモでは、以下に簡略化します。

```text
1. 職員が住民異動イベントをCSVまたは手入力で登録する
2. システムが住民異動イベント一覧に表示する
3. 職員が「資格登録へ」ボタンを押す
4. システムが資格異動登録画面を表示する
5. 職員が登録内容を確認して登録する
6. システムが被保険者番号を採番する
7. システムが被保険者を登録する
8. システムが資格履歴を登録する
9. 職員が被保険者詳細を見る
10. 職員が被保険者証をプレビューする
11. 職員が被保険者証を発行する
12. システムが証発行履歴を保存する
```

---

## デモで作る画面との対応

| 業務ステップ | 作る画面 |
|---|---|
| 住民異動情報を受け取る | 住民異動イベント登録・取込画面 |
| 対象者を確認する | 住民異動イベント一覧 |
| 資格情報を登録する | 資格異動登録画面 |
| 登録済の人を確認する | 被保険者一覧 |
| 1人の情報を見る | 被保険者詳細 |
| 証を作る | 被保険者証プレビュー |
| 発行履歴を見る | 証発行履歴 |

---

## デモで作るDBとの対応

| 業務ステップ | DBテーブル |
|---|---|
| 住民異動情報を受け取る | resident_change_events |
| 被保険者を登録する | insured_persons |
| 資格履歴を登録する | qualification_histories |
| 被保険者証を発行する | certificate_issue_histories |

---

## 実装時に大事な業務ルール

### 1. 65歳到達なら資格取得

65歳になる人は、基本的に第1号被保険者として資格取得します。

PoCでは、イベントタイプ `AGE_65` を受け取ったら、資格取得として扱います。

### 2. 転入なら資格取得

他の市区町村から転入してきた場合も、介護保険資格の取得対象になります。

PoCでは、イベントタイプ `MOVE_IN` を資格取得として扱います。

### 3. 転出・死亡なら資格喪失

市外へ転出したり、死亡したりした場合、その市区町村の介護保険資格はなくなります。

PoCでは、イベントタイプ `MOVE_OUT` と `DEATH` を資格喪失として扱います。

### 4. 住所・氏名変更なら資格変更

住所や氏名が変わった場合、資格そのものは継続しますが、管理情報を変更します。

PoCでは、イベントタイプ `ADDRESS_CHANGE` と `NAME_CHANGE` を資格変更として扱います。

### 5. 被保険者番号は自動採番

新しく被保険者を登録するとき、被保険者番号を自動で付けます。

PoCでは、10桁のゼロ埋め文字列にします。

例：

```text
0000000001
0000000002
0000000003
```

# 02-02 被保険者証等再交付

## 何の業務か

これは、

> 被保険者証をなくした、汚した、壊したなどの理由で、もう一度発行する業務

です。

---

## 本来の流れ

```text
1. 住民が再交付申請書を提出する
2. 介護保険担当課が申請を受け付ける
3. 対象者を確認する
4. 介護保険システムに再交付情報を登録する
5. 被保険者証を再作成する
6. 住民へ送付または手渡しする
```

---

## 今回のデモでの流れ

```text
1. 被保険者詳細画面を開く
2. 再交付申請画面を開く
3. 申請理由を選択する
4. 再交付申請を登録する
5. 再発行ボタンを押す
6. 被保険者証を再発行する
7. 証発行履歴に「再交付」として登録する
```

---

## 再交付で作る画面との対応

| 業務ステップ | 作る画面 |
|---|---|
| 対象者確認 | 被保険者詳細 |
| 再交付申請受付 | 再交付申請画面 |
| 再発行 | 被保険者証プレビュー |
| 履歴確認 | 証発行履歴 |

---

## 再交付で作るDBとの対応

| 業務ステップ | DBテーブル |
|---|---|
| 再交付申請を登録する | reissue_applications |
| 再発行履歴を残す | certificate_issue_histories |

---

## 実装時の注意

再交付申請と被保険者証発行履歴は分けて管理します。

理由は、

- 申請されたこと
- 実際に発行されたこと

は別の事実だからです。

つまり、DBとしては以下のようになります。

```text
reissue_applications
  → 再交付の申請情報

certificate_issue_histories
  → 実際に証を発行した履歴
```

この分け方にしておくと、将来的に「申請だけされたが未発行」「却下された」「後で発行した」といった状態も扱いやすくなります。

---

# デモで説明するときの言い方

今回の業務を説明するときは、以下のように言えば伝わりやすいです。

> 今回のデモでは、介護保険システムのうち、被保険者資格管理に範囲を絞っています。住民記録システムとの本物の連携は行わず、住民異動イベントをCSVまたは手入力で代替しています。その住民異動イベントをもとに、資格取得・変更・喪失を登録し、被保険者番号を自動採番し、被保険者証をHTML/PDFで出力し、発行履歴を残すところまでを実装しています。さらに余裕があれば、被保険者証の再交付申請と再発行も実装します。

この説明ができれば、標準仕様書を細かく読んでいない人にも伝わります。
